3) Why this architecture exists

Why: Browser-native reliability

Browser automation must happen in extension + content contexts, not a remote server. This design keeps interaction local to user tabs while preserving model orchestration in one control plane.

Why: Multi-surface control

Users need both UI-first and terminal/agent workflows. Relay + CLI layers expose the same core tooling and run lifecycle without duplicating business logic.

Why: Economic controls

Paid model routing requires authentication, usage metering, credit reservation/settlement, and Stripe lifecycle handling. Backend services isolate those responsibilities from extension runtime.

How: Contract-first messaging

Shared runtime and JSON-RPC contracts define event schemas and request/response behaviors, enabling extension UI, relay, and clients to evolve against a common protocol.

4) Fundamentals currently present in code

These are already in the codebase and should remain non-negotiable design primitives.

Session isolation + run cancellation

Active run maps, per-session state, cancellation guards, and stop semantics prevent output spill across sessions and keep behavior deterministic.

Bounded memory and payload controls

MAX_* caps, truncation, dataUrl stripping, history clamps, and report image limits are consistently used to avoid runaway memory and message payload growth.

Policy and permission gates

Tool categories, allowed-domain checks, screenshot toggles, and account mode gates establish explicit runtime policy boundaries around automation behavior.

Fallback and resilience loops

Model fallback ordering, OAuth candidate refresh, finalization retries, and safe-await wrappers prioritize run completion under imperfect provider behavior.

Streaming event architecture

Runtime event schema with stream start/delta/stop, run_status, tool events, and terminal final/error enables progressive rendering and telemetry visibility.

Shared protocol layer

@parchi/shared unifies plan/runtime/tools/json-rpc types so extension, relay, CLI, and backend integration points can interoperate with reduced drift risk.

5) How we should structure going forward

Recommended target: explicit bounded-context modules around orchestration, tools, protocol, accounts, and UI rendering so complexity is distributed by responsibility instead of accumulated in mega-files.

A. Runtime Orchestrator Layer (background)

  • run-engine/: run lifecycle, retries, finalize/compaction orchestration
  • message-router/: sidepanel + relay message handlers per command group
  • tool-runtime/: permission gate + tool dispatch + result enrichment
  • session-store/: session state, run registries, capacity eviction rules

B. Tooling Layer (browser-tools)

  • Replace switch dispatch with typed registry: toolName -> handler
  • Shared wrappers: tab resolution, overlay signaling, timeout normalization, result envelope
  • Domain files: navigation, interaction, extraction, capture/video, tabs/session

C. Sidepanel App Layer

  • Separate runtime-event reducer from DOM renderer components
  • Normalize feature modules by domain (chat, account, settings, timeline, tabs)
  • Keep prototype augmentation style if desired, but centralize shared state transitions

D. Backend & Billing Layer

  • Split payments by concern: plan-config, stripe-session, key-provision, recovery
  • Keep aiProxy focused on transport + accounting hooks; push provider policies to provider modules
  • Treat credits reservation/settlement as a strict domain service API

E. Contract Layer (shared)

  • Single source of truth for runtime events + rpc + tool schemas
  • Add versioned compatibility rules and deprecation windows for event evolution
  • Use shared validators at ingress points (UI messages, relay RPC, backend payloads)

6) Immediate execution plan

Wave 1 — dispatch simplification

Start with browser-tools.ts registry extraction and shared preflight wrappers. Low-risk, high readability, measurable line reduction.

Wave 2 — event router split

Split sidepanel handleRuntimeMessage into message map handlers grouped by run/tool/plan/ error/session events.

Wave 3 — run-engine extraction

Move model pass/fallback/finalization + compaction from service.ts into a dedicated run-engine package-local module.

Practical principle: do not change behavior first—extract and stabilize interfaces, then optimize.
